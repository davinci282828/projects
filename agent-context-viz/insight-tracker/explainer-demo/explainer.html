What this “build” is

The input folder for this run contains only instructions (no application source code). So the system you’re looking at is a documentation build: it reads ./input/ and produces a self-contained explainer.html that’s interactive, scannable, and offline.

2. Key Decisions

Expandable cards: why this is structured this way, trade-offs, and alternatives.

why + trade-offs

Single self-contained HTML output

offline

Why: the template requires no external deps and offline use. One file is simplest to distribute and review.

Trade-off: code is less modular than multi-file apps; reuse is limited.

Alternative: multi-page site + build step (bundler) — rejected by the “works offline, single file” constraint.

Template-driven, fixed section order

structure

Why: the explainer must follow a predictable mental model for fast reading (architecture → decisions → components → flow → interactivity).

Trade-off: less freedom to reorganize content per project; but faster to scan across builds.

Alternative: free-form narrative — tends to bury key points and fails the 3-minute constraint.

Lightweight “good enough” syntax highlighting

no libs

Why: external highlighters are typically large (and violate the no-deps constraint). Here we use a tiny regex-based tokenizer for Markdown-ish excerpts.

Trade-off: not a full parser; edge cases won’t color perfectly.

Alternative: ship a full syntax highlighter (Prism/Highlight.js) — rejected for size/dependency reasons.

Explain the build even when app source is missing

robust

Why: input sets can be incomplete. This explainer degrades gracefully by documenting the “documentation-build” system: inputs, constraints, and the generated UI behaviors.

Trade-off: less value than a full app walkthrough, but still provides a clear artifact and the exact expected structure.

Alternative: error out — would fail “build exactly what TASK.md describes” (which is to generate explainer.html).

5. Interactive Elements

What’s interactive in this explainer (and how it’s implemented), aligned with the template requirements.

UX
Collapsible sections for detail levels
Uses native <details> for low-cost, accessible disclosure. Decision cards also support click-to-collapse via a small JS handler.
“Jump to” navigation sidebar
Sidebar links anchor to each mandatory section and are enhanced with scroll-spy to show where you are. This keeps the explainer scannable and within the 3-minute target.
Progress indicator
A top bar in the sidebar computes scrollY / (docHeight - viewportHeight) and renders a percentage. It’s lightweight and works offline.
Click to expand code blocks
Code blocks start with a capped height for fast scanning; the Expand button toggles a CSS class on the <pre>. There’s also a Copy button to move snippets into your editor quickly.
Size discipline: interactivity is implemented with small, dependency-free JS and CSS. This stays aligned with “single HTML file”, “works offline”, and “under 150KB”.
