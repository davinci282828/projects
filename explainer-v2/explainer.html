How the counter works

This app is a tiny, single-page counter: semantic HTML for structure, CSS for the glassy card UI, and a small JS module that holds state (count) and re-renders the number on each button click.

Try it — click + and −, then read how that click updates state and re-renders the UI.
State:
let count = 0 (in JS)
UI value:
<output id="value"> (in HTML)
Update path:
click → count +/- → render()
Scope:
All logic wrapped in an IIFE (no globals)

1. Architecture Overview

The app is intentionally split by responsibility: HTML declares the controls, CSS styles them, and JS wires events to a single state variable plus a render() function.

UI • Styling • Logic
UI (markup) Styling Logic (state + events)
The only “data” is an in-memory number. The UI never reads from the DOM to compute state; it always treats count as the source of truth and uses render() to push updates into the <output> element.

2. Key Decisions

Small app, but a few choices keep it predictable: isolate state, centralize rendering, and keep the DOM contract explicit with IDs and ARIA attributes.

Trade-offs
Single state variable + explicit render() Predictable UI updates and a single place to change rendering behavior.
▾
Instead of directly mutating the DOM in each click handler (duplicating logic), both handlers update count then call render(). That’s the entire update loop.
Chosensingle render path

  valueEl.textContent = String(count);
};
Alternativeduplicate DOM updates
What it could look likeharder to evolve

  count -= 1;
  valueEl.textContent = String(count);
});

incBtn.addEventListener('click', () => {
  count += 1;
  valueEl.textContent = String(count);
});
IIFE wrapper (module-like scope) Keeps variables off window; avoids name collisions.
▾
The entire script is wrapped in (() => { ... })(), so count, render, and DOM references are private. For tiny apps, this is a pragmatic “no bundler, no globals” pattern.
Semantic + accessible output <output> with aria-live announces changes politely.
▾
Using <output> makes the number “a computed result” instead of a generic <div>. aria-live="polite" and aria-atomic="true" help assistive tech announce the updated count.

app.jsclick to expand
(() => {
  const decBtn = document.getElementById('decrement');
  const incBtn = document.getElementById('increment');
  const valueEl = document.getElementById('value');

  let count = 0;

  const render = () => {
    valueEl.textContent = String(count);
  };

  decBtn.addEventListener('click', () => {
    count -= 1;
    render();
  });

  incBtn.addEventListener('click', () => {
    count += 1;
    render();
  });

  render();
})();

4. Data Flow

When you click a button, the handler mutates count and then re-renders the UI. The steps below animate when you press “Play”, and also when you click in the preview.

Animated
Preview clicks will also trigger this (via postMessage).
1
User clicks + or −The browser dispatches a click event on the button element.
2
Handler updates in-memory statecount += 1 or count -= 1 updates the single source of truth.
3
render() writes to the DOMvalueEl.textContent = String(count) updates the <output> element.
4
UI is repaintedThe new number becomes visible; the aria-live region can announce the change.
